# Internationalization (i18n)

## Locales

- **Routing** (`src/i18n/routing.ts`): `en`, `th` are the active app locales.
- **`jp.json`**: Partial catalog for occasional copy (e.g. fallbacks in `intlMessageFallback.ts`). It is **not** required to match `en.json` key-for-key. Run `npm run i18n:check` only enforces **en ↔ th** parity.

## Adding strings

1. Add the key to `src/messages/en.json` and `src/messages/th.json`.
2. Run `npm run i18n:check` before pushing.
3. Use `useTranslations()` (or `t` from props) in components — avoid hardcoding user-facing English in JSX.

## Turbopack / RSC and “missing” keys

Next.js can serialize a **subset** of flat keys into the client `messages` object. Keys that disappear in dev still trigger `MISSING_MESSAGE` handling:

- **`src/i18n/buildMessagesForLocale.ts`** — single ordered pipeline that re-applies fragile keys from static `en.json` / `th.json` imports (see `MESSAGE_MERGE_PIPELINE`).
- **`src/i18n/intlMessageFallback.ts`** — `getMessageFallback` supplies copy when a key is still missing at runtime; it reads bundled JSON via `readCatalogString` where possible.

Do **not** add production-only user strings only to `intlMessageFallback` — prefer the JSON catalogs so translators and `i18n:check` see them.

## Development warnings

In **`NODE_ENV === development`**, `NextIntlClientProviderWithFallback` logs a **one-time warning** per distinct `MISSING_MESSAGE` so typos are visible. Production stays quiet.

## Styling note

Layout copy often uses **Tailwind** for spacing; complex widgets may use **MUI** `sx`. Prefer one system per surface when refactoring (see team conventions).
